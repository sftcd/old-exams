Question 1

 

A commercial software development group develop bespoke software, i.e. systems which are developed in response to a particular customer contract and where the customer owns the resulting IPR and source code.

 

a) Describe how this company might sensibly handle security issues for the systems they develop, with a particular focus on how this impacts on their software development processes. (10)

 

The answer ought to focus on the processes which the S/W dev company uses, and in particular in how they demonstrate to their customer that they have followed those processes. The overall completeness and correctness of the answer is worth 4 marks.

 

A sensible description of how each of the following, (or similar) fit into such a process is worth 1 mark, up to max. 6:

 

·  Formally agreeing security requirements with the customer

·  Doing an early stage risk analysis based on the functional and non-functional requirements, and making this visible to the customer

·  Including "security considerations" sections in all design documentation

·  Establishing a separate team, not including the direct developers, that reviews security issues at each stage of the development process.

·  Implementing a "secure" version of whatever source code control and design documentation versioning systems are in place (e.g. with controlled access).

·  Building up an in-house database of relevant vulnerabilities, based on the tools generally used and on the functional requirements seen in previous projects.

·  Testing specifically for vulnerabilities, e.g. using tiger teams.

·  Including security guidelines and operational security issues as part of the deliverable documentation

·  Making a formal security evaluation (ITSEC/CC) a part of the contract

·  Putting in place a way to distribute future security patches in addition to normal support/bug-fixing

·  Offering a service to continually review security given that 3rd party components (e.g. DBMSs) change over time.

 

b) If you were a customer of the above software developer, describe in general terms how you would test the security of the system they have delivered (10).

 

The main focus here should be on how the customer should insist on visibility of the relevant security documentation and test results at each stage of the process, but also noting that the customer must put in place some kind of additional security testing/review, whether this is done by the customer's own staff or yet another consultant. The overall completeness and correctness of the answer is worth 4 marks.

 

A sensible description of how each of the following, (or similar) fit into such a process is worth 1 mark, up to max. 6:

 

·  Making security requirements part of the request for proposals/quotation process so that potential developers have to address security before they start

·  Carrying out an analysis of potential security issues prior to writing the main system requirements

·  Insisting on familiarity with the testing tools and methods used by the developer.

·  Insisting that significant tests carried out by the developer are reproducible

·  Insisting that the developer include test methods and results as deliverables

·  Putting in place someone to check the delivered test methods & results

·  Putting in place a "staging" system where s/w deliverables can be installed and tested without affecting the "live" system

·  Running a late-stage risk/vulnerability analysis as part of acceptance testing

·  Checking public vulnerability databases for relevant entries and querying the developer as to the whether they apply to the system in question.

 

c) Briefly describe a specific vulnerability which is hard to detect in such an environment, and justify your selection. (5)

 

Any reasonable matching vulnerability gets marks from 5, with higher impact, less obvious ones being better rewarded. Possible things here could be:

 

·  Operating system and DBMS version-specific vulnerabilities are easily missed since it can be very hard to exactly get the same runtime environment set up in the s/w developer's network, e.g. its not easy for the s/w developer to ensure that no DBMS admin account for the live system still uses a default password

·  Similar to the above, parts of the application may be unexpectedly deployed inside the DMZ on the customer network, e.g. if a web server application needs access to a corporate LDAP then the directory may be exposed to malware running in the DMZ.

·  The s/w developer may not really be familiar with operational issues on the customer site, so e.g. a single employee may end up taking on what the s/w developer thought would be multiple roles, thus making certain attacks easier, e.g. if a single systems administrator handles both the OS and the application then that person can probably misuse the application and scrub some logfiles.

 

Question 2

 

a) Briefly describe two vulnerabilities which are different in an embedded systems context, compared to the typical Internet/web application. (5)

 

2 points for each being correct, with 1 additional point for interesting choices.

 

Examples:

 

·  Recovery from DoS attacks can be harder in embedded systems since they are more likely to be remote and sometimes much harder to reboot, however, embedded systems are generally less vulnerable to DoS since they are usually easier to test, having fewer meaningful interfaces. Resource-consuming DoSes may also have more impact in embedded systems since resources are typically more scarce.

·  Badly chosen passwords are a common problem in non-embedded systems and can be serious given that the same password is used for many different applications, so e.g. an IMAP account password could also be used for Internet banking. In embedded systems there are generally fewer accounts (or none) though bad password values and ongoing use of defaults occur more commonly.

 

b) Name a common countermeasure which is applicable for both typical Internet/web applications and embedded systems. Briefly describe how the implementation and/or deployment of that countermeasure might differ in these two contexts. (5).

 

For picking a relevant countermeasure, 1 mark, for a correct description of  that countermeasure 1 mark, for describing interesting differences 2 marks and one additional mark for overall quality.

 

If they picked SSL, which is probable, then its description is obvious. SSL implementations require crypto and the libraries and general support available on embedded systems isn't a good for that, and crypto accelerators may not be available to improve performance as they are for standard server hosts. SSL involves a few roundtrips to establish a session which could be a problem in some bandwidth or communications challenged environments. SSL uses PKI which uses revocation/OCSP which may be problematic in embedded systems. Maintaining state for many SSL sessions could be hard for an embedded system. Installation and commissioning of private keys and certificates for SSL is different too.

 

c) Describe how X.509 based public key infrastructure can support security for embedded systems. (5)

 

The main thing here is to note that some types of certificate content are more suited for embedded systems than others. There is also the major trick of not placing the certificate onto the embedded system (just a reference, e.g. a URL) which can solve many problems (e.g. with expiry/renewal). Using XKMS or SCVP instead of LDAP/CRLs/OCSP may also be better. Correct discussion of these or equivalent factors is worth 2 points each, with one more for overall quality.

 

d) Describe two difficulties in deploying X.509 based public key infrastructure in embedded systems. (5)

 

There're many from which to choose! So 1.5 each with 2 more for overall quality. Possibles include:

 

·  CRL sizing issues

·  Access to OCSP responders

·  Certificate chains can bloat protocols beyond the communications capabilities of some systems

·  Root CA installation/update is harder in these environments

·  Practical on-board private key generation may not be possible due to CPU limitation

·  The system may not have a real time clock, thus making it hard to handle revocation and expiry.

 

e) Give an example of how using cryptography affects performance on low-end embedded systems and briefly describe one way in which this can be handled. (5)

 

Example is worth up to 3 marks (one for a correct example, one for a good description and one for a bonus). One mark for ways to handle and one for another bonus.

 

RSA based digital signing or decryption requires expensive private key operations and may be very hard on e.g. an 8-bit CPU. This is due to the way that RSA generally uses shorter public exponents for validation/encryption. This could be handled using the Chinese remainder theorem speed-up, or by splitting the calculation over more than one host (worth a bonus if that's mentioned!), or even by falling back to MACs where symmetric key management can be done. Simply moving the signing/decryption to another host in the network is also sometimes possible. Lastly, one can include specific crypto accelerator chips even in small embedded systems (e.g. smartcard processors) which handle the crypto well even if at a cost. Such chips can also include tamper resistant storage for keys which can be a side-benefit.

 

Question 3

 

You are a developer responsible for embedded systems security in a consumer products development company. The company are developing a range of kitchen equipment (kettles, cookers, toasters, fridges etc.) which can be monitoring remotely, and in some cases, activated remotely, via IP networking based on an RF link layer (e.g. IEEE 802.11b or Bluetooth) . The equipment in the kitchen has to require zero-configuration, so for example, IP address configuration has to happen transparently and automatically when the equipment is first installed. If there is a base station in the home, then the equipment will detect this and route packets via the base station. The base station is a separate product from the same company, and can provide a range of security services.

 

a) Describe the potential safety and privacy considerations applying to this overall system. (10)

 

Safety and privacy get 3 marks each here with 2 for the overall answer and 2 more for considering the likelihood and impact for each (i.e. taking a sound risk analysis approach).

 

Possible issues, (one mark each):

 

·  Accidental unexpected operation (fire etc.)

·  Deliberate attack aimed at continually operating e.g. a kettle until its dry and goes on fire, turn on hob when a flammable item is likely to be on top

·  Address collisions could result in accidental unintended operation ("turn on kettle"=>"turn on hob" and that while I'm sitting on the hob)

·  Tuning off the fridge for a short while so as to spoil food aiming at poisoning.

·  The base station could be DoSed (or otherwise fail) preventing me from turning off equipment

·  If I were running one a cable network or wireless broadband then my neighbours might be able to see which equipment I'm using when

·  Traffic to DHCP servers or similar may reveal which types of equipment I own

·  The equipment may act as spyware, in particular clandestinely revealing ownership and usage information to the manufacturer; this would be more likely if the base station is used.

·  If the fridge scans barcodes then it could expose further product preferences, e.g. a bottle of antibiotic might determine the illness from a family member is suffering.

 

b) Briefly describe three other vulnerabilities (not related to safety or privacy) which apply in this context. (5)

 

One mark each, plus two for overall.

 

·  Buggy kitchen equipment might affect other parts of the home network, e.g by poor address choices or badly implemented IP networking, or vice-versa for other home network equipment (e.g. a TV).

·  The base station might include vulnerabilities which turn it into an attack vector allowing access into the home network (e.g. if the base station is accessible from the Internet and has some buffer overruns).

·  If the fridge can send orders to the supermarket, then hacking the fridge can result in transfer of funds (fridge sends instruction to buy a phoney phone card product).

·  If I can route packets (even indirectly) to equipment from the Internet then I may be able to use that equipment as a DoS attack vector against some other Internet host (e.g. by spoofing the source address, or requesting continual status reports to a victim address).

·  If the equipment has any stored secrets then these are likely to use default values and so it may be possible to deny service by using that secret to place the equipment into a non-functioning state from which the user cannot recover.

 

c) Describe how you would mitigate each of these vulnerabilities. (10)

 

Three marks for each, plus one overall.


Taking the 1st three above:

 

·  Ultimately you really need to have some way to debug your home network and/or some IDS type functionality on the network. Partitioning off different parts of the network from one another would also be sensible and could be achieved with a firewall (which is likely to be there in any case). I'd also require some implementation standards for such equipment since it’s a consumer product and not really a software product.

·  The base station s/w really has to be tested. Again, firewalling off the base station from the rest of the network will help, or perhaps putting the base station into a DMZ and only allowing equipment related packets to/from the base station. Using some form of authentication especially on packets arriving at the base station from the Internet would help, probably via an SSL connection (which could also prevent exposing you base station password to a fake base station).

·  The supermarket should maintain a list of what the fridge is allowed to order and should not accept random orders. Only products which don't allow the attacker to directly benefit financially should be allowed to be auto-ordered. There should be an option for the owner to approve orders prior to them being sent.

 

Question 4

 

You are responsible for developing a web-based conference management system which handles the call-for-papers, paper submissions and reviews, the conference agenda and registrations. When an attendee registers they are passed onto a separate billing system which is operated by a 3rd party. There are similar links to hotels nearby the conference venue which accept reservations over the web. Authors of accepted papers do not have to pay the conference fee, and there are special reduced rates for students.

 

For a typical conference this is all fairly simple. However, this system is to be specially designed for running hacker conferences and (as a result of previous incidents!) the conference organisers have widely publicised the fact that the new system will be "secure".

 

a) Give an overview of the system, (e.g. including a network diagram) emphasising how the system differs from others with less stringent security requirements. (5)

 

Diagram should show web server, payments server, DB etc. A good diagram is worth 3 marks (2 if no DMZ/firewalls shown).

 

The system should use SSL where appropriate, e.g. between the site and payments gateway.

 

b) Describe what you think are the four most-important threats and how you would counter each of them (10)

 

2 marks each, and 2 for the overall.

 

Some threats, just 4, but many more could be given:

 

·  DoS on the web server; this is the most likely, and can probably best be gotten around by contracting out operation of the web server front-end to a service provider who can provide the resiliency and redundancy required to defeat DoSes.  Requiring that the provider run the web server on more than one OS etc. Also need to make sure the service provider knows when critical periods occur, e.g. the day paper submission (really!) closes.

·  DoS on the payments server or payment server DB hacked, less likely and lower impact but still stops registrations being processed; main counter is to hand-off payments processing to a large, reputable outfit, and keeping them informed as above.

·  Conference DB hacked for free attendance; ensuring that only the web servers can talk to the DBMS by using appropriate DMZ, f/w and IDS etc. Making sure the DB is backed up frequently. Locking DB records as an when possible (e.g. preventing write access to submitted papers tables once submissions are closed).l

·  Auto-Submission of many spoof papers, perhaps so that they're not obvious spoofs as a DoS against the reputation of the claimed authors and the conference (if one gets through). Main counter here is to close-the-loop before accepting a submission by getting an email to/from the claimed main author (perhaps notifying the other authors too). Can also help if submission is somewhat interactive and say uses the type of password-in-GIF approaches used to prevent spammers subscribing to lists/community sites. Running plagiarism-detecting s/w on submissions may be required to spot re-submission of modified papers already accepted elsewhere.

 

c) Describe another relatively improbable, but highly damaging vulnerability and how you would counter that (note: this must not overlap with the previous part) (5)

 

This is a straight marks-from-5, with credit for inventiveness! Has to be improbable but also have impact though, or no/few marks at all.

 

For example:

 

·  Hacking the on-site registration systems' printers (and I mean the printers, not the print server) so that badges randomly print rude remarks. Main counter is pre-printing (most) badges and/or using serial printers and ensuring that the print server runs some f/w, IDS, etc code.

·  Mounting an NTP attack against the web server and DB hosts so as to move the clock back to allow some now-expired action to occur, e.g. late submission of a paper describing this attack in detail.

·  Mounting a content-format based (e.g. JPEG or Postscript or whatever other formats are available) attack on the reviewer's systems using the submitted paper file as the attack vector. Maybe have the attack code only accept this paper and nothing else. Run submitted papers through some virus-scanner or equivalent.

·  Direct an attack against the conference chair's work PC so as to gain full change control on the conference site and DB and then modify at will. Ensure that the conference chair uses a hardened PC & OS. Where possible ensure that the chair plus a randomly selected PC member has to approve all actions (e.g. when chair does something potentially bad, mail a notification to a PC member and require an email ACK).

·  Launch a phishing attack against all the PC members to gain write access to the system and leak out the "PC-only" review comments (plus some scurrilously invented ones!). Counter here is PC member education: assume them that you'll never send such a web based form to them for any reason.

 

d) Given all of the above, if you were to attack the system, how would you go about launching such an attack? (5)

 

This is a straight marks-from-5, with credit for inventiveness!

 

Hmmm…I might:

 

·  Create phoney conference web sites with similar URLs (e.g. typosquatting) and passively await good username/pwd combinations to arrive – then sell on the ability to get in for nothing.

·  Create a phoney competing conference web site and marketing campaign and at a bad moment (e.g. CFP issue, end of early-bird registration) blackmail the real conference organisers by threatening to "go live" with the competing conference.

·  Well in advance of the conference mount an attack against the hotel/conference centre's network infrastructure inserting a trapdoor to be executed during the target conference. The trapdoor might filter for interesting packets/flows (e.g. passwords or defamatory remarks) and forward/publish those.

Question 6

 

a) Describe how, and more importantly, why, hash functions are used in a real key management system (e.g. Kerberos, TLS or S/MIME). (10)

 

Hash functions are used to identify things which ought to be related (e.g. a previous exchange, since digesting derives a fixed length, checkable, hard to predict, unique identifier for any set of bytes), to derive new things (in particular in keys via KDFs, similar reasons, though also as a way to squash entropy into key-sized chunks) and for digitial signing (within rsaWithSha1, to prevent e.g. RSA number-based attacks) and MACing (e.g. within HMAC). Hashes have also been used at application layer to identify e.g. s/w being distributed.

 

Identifying each function gets 1 marks. Saying why for each gets another up to 8. 2 more for the overall.

 

b) Briefly describe each of the three main security properties which are desirable for hash functions. (5)

 

Amounts to a request to define pre-image, 2nd-pre-image and collision reisistance for 1 mark each with an additional 2 overall given for:

 

·  mention of the different complexities involved

·  explaining the birthday paradox

·  mentioning other algorithms (eg. MD2, MD4, Whirlpool) and their status

·  mentioning efficiency and how its more important for digests than encryption

 

If they get it wrong but mention other desirable features (e.g. ramdom looking output) which are less hash-specific, then those will get 0.5 marks each, if correct.

 

c) What are the effects of the recently discovered weaknesses in hash functions? (5)

 

Straight marks from 5 here.

 

Main effect is that collisions have been found for MD5 and hinted at for SHA1, and e.g. two colliding X.509 certificates have been demonstrated as a result (based on an MD5 collision). However, the SHA-1 weakness is only claimed to reduce the work from 2^80 to 2^69 which is still better than MD5 ever was, so the attack isn't a killer (yet). The fact that SHA256 etc are also based on the same family of constructs as MD5 and SHA-1 means that it may be wise to go looking at other constructs at this point. There is currently no known effect if the application data which is being digested is unpredictable, which is the case in most contexts, with the major exception of X.50 certificates.

 

d) Describe two issues which might arise in migrating X.509 based public key infrastructure implementations from use of MD5 or SHA-1 to for example SHA-256 as was planned to happen in the 2010 timeframe. (5)

 

2.5 marks each; Possible issues:

 

·  You'd need to be able to do bulk certificate roll-over

·  Client and server s/w deployment takes about 5 years

·  Hard to ever totally remove an algorithm since someone'll always have an application that cannot be updated.

·  Root CA certificates exist with very late expiries (2020+) which will ostensibly remain valid and thus remain as potential vulnerabilities until those dates esp if 2nd-pre-image resistance falls.
